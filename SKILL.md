## 配置

`config.json` 中配置测评根目录：

```json
{
  "测评根目录": "C:\\AI编程项目\\字节coding众测"
}
```

每次启动时读取此文件获取根目录路径。如路径变更，直接改 config.json。

## 模式判断

本 Skill 支持两种评测模式，步骤1读取任务书时自动判断：

| 判断条件 | 模式 | 评分体系 |
|----------|------|----------|
| 批次文件夹名或任务书包含"openclaw"/"open claw" | OpenClaw 模式 | 产物质量⭐ + 交付效率⭐ + 综合表现 + 轨迹圈选 |
| 用户在对话中说了"openclaw" | OpenClaw 模式 | 同上 |
| 其他情况 | Coding 模式 | 需求(0-5) + 代码(0-3) + 轨迹(0-2) + 交付效率 |

**两种模式下步骤1-5完全一样**，只有步骤6（评分）和步骤7（报告模板）有区别。

OpenClaw 模式下，评分不拆固定维度（因为不一定是代码任务），而是根据任务书要求和实际产物综合判断。评分依据来自：自动化测试结果 + 用户截图（文件名=评价）。

## 主流程

### 步骤1：任务设计（准备阶段）

**触发**：用户说"准备测评"、"设计任务"，或新一批测评开始时

1. 读取 `references/评测标准.md`、`references/流程优化经验.md`
2. 与用户确认本批次日期，创建批次目录（如 `{测评根目录}/{YYYY-MM-DD}方舟coding众测/`）
3. 读取 `{批次目录}/任务书.md`。如文件不存在，提醒用户从平台复制任务书内容到该路径
4. 与用户确认：
   - 本批次有几个项目，每个项目名称
   - 每个项目基于哪个已有代码仓库
   - 本次测试的模型名称列表（从任务书中读取）
5. 创建完整批次目录结构（输出到对话，用户确认后再创建）
6. 协助用户撰写每个项目的任务说明（提示词），遵循以下原则：
   - **不做从零到一**，只在已有项目上新增一个功能
   - 复杂度控制在模型跑 20-50 轮工具调用的范围
   - 提示词里写清楚验证目标（方便打分）
   - 不给参考项目链接（模型没有联网能力）
   - 写清楚任务背景和技术栈
   - 列出具体的验证目标（1234 逐条，"点击XX → 看到YY"形式）
   - 写清楚技术约束
   - 选择标准：浏览器里能直接看到效果、本地能跑不依赖外部环境、验证步骤明确
7. 如有已有代码，协助打包上传（排除 node_modules、dist）
8. 用户在平台提交任务后，等待模型运行（约30分钟-1小时）
9. **生成并行会话启动提示词**（见第七节模板）

**功能类型参考**（选题时参考）：

- 注册登录 + 访问控制
- 前后端分离改造（把单 HTML 拆成前后端）
- 小程序转 Web/PWA
- 接入第三方 API（需提供可用的 API Key）
- 数据持久化（加数据库，实现增删改查）
- 文件上传/下载/导出（CSV、Excel、图片等）
- 搜索 + 筛选 + 分页
- 评论/留言/收藏系统
- 多步骤表单 + 复杂表单验证
- 拖拽排序/看板交互
- 暗黑模式/主题切换/国际化
- 数据可视化（图表、仪表盘）
- 权限管理（角色区分、菜单控制）

**完成标记**：`echo "TASK_DESIGNED" >> scripts/.check_status`

**步骤完成时在对话中输出**：`[✓ 步骤1完成] 任务设计完成，已创建目录结构，任务说明已写入。等待平台模型运行。`

### 步骤2：下载产物

**触发**：平台模型运行完毕（约30分钟-1小时后）

下载由用户操作，AI 无法操作。

1. 提醒用户从平台下载各模型的产物代码（zip）
2. 用户将 zip 放到对应模型文件夹
3. 等待用户确认"下载好了"
4. 未就绪的模型标记"待补"，不阻塞流程

**完成标记**：`echo "DOWNLOAD_DONE" >> scripts/.check_status`

**步骤完成时在对话中输出**：`[✓ 步骤2完成] 用户确认产物已下载到位。`

### 步骤3：解压或确认已就绪

**触发**：用户确认下载完毕，或用户说"XX模型好了"

1. 如果用户提供了 zip 文件，AI 自动解压到对应模型文件夹（`unzip -o`）
2. 如果用户已自行解压，直接确认目录结构
3. 检查解压后的项目结构，识别前后端目录
4. 输出解压结果到对话供用户确认
5. **不需要等所有模型就绪**：已就绪的先处理，后续模型就绪后随时从步骤4补充
6. **生成并行会话启动提示词**：所有项目解压完成后，为每个项目生成独立会话的启动 prompt（见并行会话 Prompt 模板）。用户拿着 prompt 去开新会话，每个会话从步骤4开始独立执行。

**完成标记**：`echo "UNZIP_DONE" >> scripts/.check_status`

**步骤完成时在对话中输出**：`[✓ 步骤3完成] 解压完成，已生成并行会话提示词。`

### 步骤4：启动服务

对每个已就绪的模型：

1. 查看项目结构，尝试识别启动方式（README、package.json scripts、Makefile 等）
2. 如果能判断怎么启动就直接启动；判断不了就问用户
3. 启动过程中遇到依赖缺失、环境问题等，问用户怎么处理
4. **预填配置**：扫描项目中需要填写的配置项，自动填好：
   - API 代理地址 → 指向对应后端端口
   - AI 相关的提示词/Prompt（如导演模式提示词、系统提示词等）
   - API Key（从环境变量读取或问用户）
   - 数据库路径
   - 其他需要手动填写才能跑的配置
   - 总之：让用户拿到手就能直接用，不需要自己去改配置
5. 端口分配：检查占用，同任务内按模型顺序递增，不同任务用不同范围
6. 启动服务，确认可访问
7. **每启动一个模型后，立刻在对话中输出访问地址**（含前端和后端地址）
8. **所有模型启动完毕后，输出汇总地址表**，格式如下：

```
## 访问地址汇总

| 模型 | 前端地址 | 后端地址 | 状态 |
|------|----------|----------|------|
| 模型1 | http://localhost:3001 | http://localhost:4001 | ✅ |
| 模型2 | http://localhost:3002 | — | ✅ |
```

9. **验证端口和模型匹配**：确认每个端口对应的确实是那个模型的代码，不要搞混。验证方式：curl 访问后检查页面标题或 JS 入口文件内容

**异常处理**：

- 启动失败 → 记录错误，尝试修复一次，仍失败则问用户或标记"无法运行"
- 端口冲突 → 换下一个空闲端口

**注意**：不需要等所有模型就绪才开始。已就绪的模型立即启动，后续模型就绪后随时补充。每次补充新模型后，重新输出完整的地址汇总表。

**步骤完成时在对话中输出**：`[✓ 步骤4完成] 已启动模型服务，访问地址如上。你可以开始在浏览器验证，同时我会执行步骤5的自动化测试。`

**完成标记**：`echo "SERVERS_STARTED" >> scripts/.check_status`

### 步骤5：手动测试引导 + 自动化测试（同步进行）

**手动测试引导**（服务启动后立刻输出）：

```
## 手动测试清单 — {项目名}

### 各模型访问地址
- 模型A：http://localhost:XXXX
- 模型B：http://localhost:XXXX

### 验证步骤（根据任务说明的验证目标逐条生成）
1. [ ] 未登录访问主页 → 预期：跳转登录页
2. [ ] 注册新账号 → 预期：注册成功自动进入主页
3. [ ] ...

### 边界场景
1. [ ] 空输入提交
2. [ ] 异常操作
```

**自动化测试**（与用户手动测试同步）：

1. 读取 `{项目名}-任务说明.md` 中的验证目标
2. 逐个模型逐条验证：curl 接口、检查代码结构、对比验证目标
3. 标注代码问题位置（`文件:行号` + 具体代码）
4. 输出 `{项目名}-自动化测试结果.md`

用户发现问题截图放到模型文件夹（**文件名=问题描述/反馈评价**）。

**重要**：自动化测试结果仅作参考。与用户截图冲突时，以截图为准。

**完成标记**：`echo "TEST_DONE" >> scripts/.check_status`

**步骤完成时在对话中输出**：`[✓ 步骤5完成] 自动化测试完成，手动测试清单已输出，等待用户截图。`

### 步骤6：评分

**触发**：用户确认手动测试完成、截图已放好

1. **读取用户截图**：必须识别图片内容 + 读取文件名（文件名=问题描述/反馈评价）
2. **读取平台执行数据截图**：用户会在模型文件夹放一张平台截图，包含耗时、用户轮次、工具轮次等数据
3. 综合截图 + 自动化测试，逐模型评分

#### Coding 模式评分

4. 完成质量评分流程（参考 `references/评测标准.md`）：
   - 先选档位（9-10/7-8/5-6/3-4/0-2）
   - 看封顶规则
   - 三维度落点：需求(0-5)、代码(0-3)、轨迹(0-2)
5. 交付效率评分（独立维度，10分制，可选）：
   - **仅当多个模型都实现了类似效果时才需要填写**
   - 数据来源：平台截图中的耗时、用户轮次、工具轮次
   - 多模型之间相对比较，轨迹越短/用户轮次越少的效率越高
   - 如果完成质量差距大（有的能用有的不能用），不需要填交付效率
   - 用户在对话中确认是否需要填写
6. **评分后逐条对照检查**（必须在对话中输出确认）：
   - [ ] 截图与自动化测试冲突 → 已以截图为准
   - [ ] 封顶规则逐条检查：核心未完成<=6 ✓/✗、无法运行<=4 ✓/✗、跑题<=2 ✓/✗
   - [ ] 分差检查：最高分 - 最低分 >= 2 ✓/✗
   - [ ] 每个维度都有理由 ✓/✗
   - [ ] 轨迹质量：是否有"宣称完成但实际不能用"的情况 ✓/✗
   - [ ] 交付效率分已填写 ✓/✗
7. 在对话中输出："评分检查完毕，各项规则均符合要求"或标注哪项不符合需要调整

#### OpenClaw 模式评分

4. **产物质量** ⭐ 1-10（支持半星）：
   - 不拆固定维度，根据任务书的具体要求和实际产物综合判断
   - 评分时对照提示词里的每个要求，看完成了多少、质量如何
   - 写一段大白话评论，说具体哪里好、哪里不行、缺什么
5. **交付效率** ⭐ 1-10（支持半星）：
   - 数据来源：平台截图中的耗时、用户轮次、工具轮次
   - 评价模型做事是否高效：一次到位还是反复绕弯、工具调用合不合理
   - 写一段大白话评论
6. **综合表现** 1-10（单选，对应平台档位）：
   - 结合产物质量 + 轨迹表现 + 交付效率，给一个综合判断
   - 档位参考：1=没有任何产出不合格 / 2-3=有产出但不能用 / 4-5=能看到方向问题多 / 6=基本合格可用但有提升空间 / 7-8=完成度好小瑕疵 / 9=几乎完美 / 10=完美的产出
   - 写一段大白话评论
7. **评分后检查**（必须在对话中输出确认）：
   - [ ] 截图与自动化测试冲突 → 已以截图为准
   - [ ] 分差检查：产物质量最高-最低 >= 2 ✓/✗
   - [ ] 综合表现最高-最低 >= 2 ✓/✗
   - [ ] 每个评分都有评论理由 ✓/✗
8. 在对话中输出："评分检查完毕，各项规则均符合要求"或标注哪项不符合需要调整

**完成标记**：`echo "SCORING_DONE" >> scripts/.check_status`

**步骤完成时在对话中输出**：`[✓ 步骤6完成] 评分完成，封顶规则✓、分差>=2✓、各维度理由✓、交付效率✓。`

### 步骤7：生成测评报告

写入路径：`{测评根目录}/{批次文件夹}/{项目名}/{项目名}-测评结果.md`

注意：文件名必须是当前项目名，不要搞混。如当前是项目A，就写入 `{项目名A}-测评结果.md`。

**格式规则（通用）**：
- 所有列表用 `1、` `2、` 中文顿号序号，不用 `- ` 或 `1.`
- 排名用 `1、` `2、` 序号
- 评论用大白话，不要AI味，说具体问题
- 轨迹圈选：文件名:行号用正文（不加反引号），代码片段用代码块，说明用正文

#### Coding 模式报告模板

打分理由拆成「需求 / 代码 / 轨迹」三块，只写用户体验层面的大白话，不出现文件名、行号、代码。
轨迹圈选才写代码位置。两者职责分清：打分理由说"什么坏了"，轨迹圈选说"坏在哪行代码"。
轨迹圈选**必须有**，每个模型至少一条。没有缺点就写亮点（好的代码实现、安全处理等）。
详细格式参考 `references/测评结果示例.md`。

~~~~

| 模型 | 总分 | 需求 | 代码 | 轨迹 | 封顶 | 效率 | 一句话 |

| 模型X | X/10 | X/5 | X/3 | X/2 | 是/否 | X/10 | 一句话总评 |
| 模型Y | X/10 | X/5 | X/3 | X/2 | 是/否 | X/10 | 一句话总评 |

排名：
1、模型X N/10
2、模型Y N/10

---

## {模型名}

### 评论（截图证据）

1、截图文件名里的内容，直接搬过来
2、...

### 评分

总分 X/10：需求 A/5；代码 B/3；轨迹 C/2。封顶：是/否（原因）。

需求：
1、...（大白话，对着截图写，不出现文件名和行号）
2、...

代码：
1、...
2、...

轨迹：
1、...

## 多会话协作

每个项目对应一个独立会话，不在同一个会话中测评多个项目。

步骤1-3在当前会话完成（任务设计、下载、解压），步骤3结束时生成各项目的并行会话 prompt。用户拿 prompt 开新会话，每个新会话从步骤4开始独立执行到步骤7。步骤8汇总在最后任意一个会话完成。

### 会话分工

| 会话         | 负责步骤 | 职责                                     | 输出                      |
| ------------ | -------- | ---------------------------------------- | ------------------------- |
| 主会话       | 步骤1-3  | 任务设计、下载确认、解压、生成并行prompt | 目录结构 + 并行会话prompt |
| 会话 A       | 步骤4-7  | 项目A（启动+测试+评分+报告）             | 自动化测试结果 + 测评结果 |
| 会话 B       | 步骤4-7  | 项目B                                    | 同上                      |
| 会话 C       | 步骤4-7  | 项目C                                    | 同上                      |
| 最后任意会话 | 步骤8    | 汇总                                     | 方舟测评汇总-{日期}.md    |

### 上下文共享

通过项目文件夹共享，不需要手动复制粘贴。各会话读写同一个目录结构：

- 会话 A 写入 `{项目名A}-自动化测试结果.md` → 其他会话可直接读
- 用户放的截图（文件名=问题描述）→ 对应会话读取文件名获取问题信息
- 所有会话共享同一个批次文件夹，最终汇总时读取各项目的测评结果

## 沟通规范

- 简洁直接，不写代码审查报告
- 评价用大白话，说具体问题
- **截图说了算**，不质疑用户观察。用户截图文件名里写的就是反馈评价
- **必须读取用户截图内容**，识别图片中的UI状态
- 分数敢拉开差距，不和稀泥
- 圈选评论：文件名:行号用正文（不加反引号），代码片段用代码块，说明用正文
- 路径不确定时主动问用户确认
- **列表格式**：所有输出内容不用 Markdown 无序列表 `- ` 和有序列表 `1.`，统一用中文顿号序号 `1、` `2、` `3、`（Markdown 列表复制到评测平台渲染异常）

## 打分注意事项

- 打分必须有区分度，不能所有模型分数趋同（最高-最低 >= 2）
- 封顶规则严格执行：核心需求未完成 ≤ 6，不可运行 ≤ 4，跑题 ≤ 2
- 每个维度都要写理由
- 用大白话写，不要有 AI 味
- 轨迹质量重点看：模型是否宣称完成了实际没完成的功能（未验证却宣称完成 = 0 分）
- 评分完成后必须在对话中输出逐条检查确认结果
