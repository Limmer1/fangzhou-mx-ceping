# 流程优化经验（历次测评踩坑总结）

每次评测开始时必读此文件。

## 1. 截图是真理

问题：自动化测试说功能异常，但用户截图显示正常。
规则：用户截图与代码分析/自动化测试冲突时，永远以截图为准。
原因：浏览器渲染、异步加载、CSS动画等可能导致自动化误判。

## 2. 分数必须拉开

问题：AI倾向给相近分数（如6-6-7），评测无区分度。
规则：
- 最高分与最低分差 >= 2
- 主动寻找差异点，不和稀泥
- 真的接近时看细节：错误处理、边界情况、UI细节
反面：三个模型都给6分 = 评测无意义
正面：5-6-8 = 清晰区分

## 3. 描述要简洁

问题：AI写出代码审查报告式长文。
规则：
- 每维度评价2-3句话
- 用大白话，不用"架构优雅"这类空话
- 说具体问题，不说抽象评价
- 好："修改密码按钮被遮挡点不了"
- 差："该模型在用户认证模块的实现上存在一定的安全隐患"

## 4. 封顶规则容易漏

问题：按维度打完分后忘记检查封顶条件。
规则：
- 评分后必须逐条检查封顶规则
- 特别注意：需要手动改配置才能跑 + 没给说明 = 触发"无法运行且无指引"
- 封顶后在报告中明确标注

## 5. 轨迹质量看验证行为

问题：把"代码逻辑正确性"当轨迹质量评。
规则：
- 轨迹质量只看：模型是否验证了UI效果
- 关键判断：模型宣称完成的功能，实际能不能用
- 宣称完成但实际不能用 = 0分（未验证却宣称完成）
- 有验证痕迹但不完整 = 1分
- 不是评代码写得好不好（那是代码质量的事）

## 6. native模块处理

问题：better-sqlite3等native模块在Windows报错"not a valid Win32 application"。
规则：
- 先尝试 `npm rebuild`（重新编译native addon）
- 不要删node_modules重装（浪费时间且可能更多问题）
- rebuild不行再检查node-gyp / VS Build Tools
- 模型依赖native模块但没给安装说明 = 扣分依据

## 7. 配置预填

规则：
- 启动前扫描 vite.config / .env / config 文件
- 常见配置自动填入：
  - PORT → 按分配的端口
  - API代理地址 → 指向对应后端端口
  - DATABASE → SQLite用相对路径
- 不确定的配置 → 问用户

## 8. 服务启动顺序

规则：
1. 先后端（等端口可用）
2. 再前端（依赖后端API代理）
3. 验证前端可访问
4. 前端报API连接错误 → 检查vite proxy是否指向正确后端端口

## 9. 跳过未就绪模型

规则：
- 已下载的立即处理，未就绪的标记跳过
- 用户说"XX好了"时从解压步骤补充
- 补充后重新做分差检查
- 不要因为等某个模型而阻塞整个流程
